# WildGuard AI Oracle DB 설정 문서

## 1. 문서 개요

본 문서는 WildGuard AI 프로젝트에서 Oracle XE 데이터베이스를 설정하고, 서비스에 필요한 테이블을 생성하는 방법을 정리한다.

현재 WildGuard AI의 1차 ML 기능은 AI Hub 야생동물 라벨링 JSON 메타데이터를 기반으로 위험 점수 `risk_score`를 예측하는 회귀 모델 API로 구현되어 있다.

따라서 DB에는 단순 위험 등급뿐 아니라 예측 위험 점수, 위험 등급, 입력값, 모델 성능 지표, 원본 요청/응답 JSON을 함께 저장할 수 있도록 설계한다.

관리자 계정으로 Oracle에 접속한 뒤 실행한다.

```sql

CREATE TABLE USERS (
```

## 7. RISK_PREDICTION_LOGS 테이블

### 설명

위험 점수 예측 API 실행 결과를 저장한다.

현재 API는 회귀 모델 기반으로 `predicted_score`를 반환하므로, DB에도 예측 점수와 위험 등급을 함께 저장한다.

```sql
CREATE TABLE RISK_PREDICTION_LOGS (
    LOG_ID NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    USER_ID NUMBER,

    DAY_TYPE VARCHAR2(20) NOT NULL,
    CAMERA_TYPE VARCHAR2(20) NOT NULL,
    WEATHER VARCHAR2(30) NOT NULL,
    LOCATION_NAME VARCHAR2(100) NOT NULL,
    TIME_ZONE VARCHAR2(30) NOT NULL,
    SEASON VARCHAR2(30) NOT NULL,
    SPECIES VARCHAR2(50) NOT NULL,

    OBJECT_COUNT NUMBER NOT NULL,
    MAX_BBOX_AREA_RATIO NUMBER(10, 6) NOT NULL,
    AVG_BBOX_AREA_RATIO NUMBER(10, 6) NOT NULL,

    PREDICTED_SCORE NUMBER(6, 2) NOT NULL,
    RISK_GRADE VARCHAR2(20) NOT NULL,
    RISK_LEVEL VARCHAR2(20),
    MODEL_TYPE VARCHAR2(30) DEFAULT 'regression',

    MESSAGE VARCHAR2(1000),
    MAE NUMBER(10, 4),
    RMSE NUMBER(10, 4),
    R2_SCORE NUMBER(10, 4),

    REQUEST_JSON CLOB,
    RESPONSE_JSON CLOB,
    CREATED_AT TIMESTAMP DEFAULT SYSTIMESTAMP,

    CONSTRAINT FK_RISK_USER
        FOREIGN KEY (USER_ID)
        REFERENCES USERS(USER_ID)
);
```

### 주요 저장 항목

| 컬럼              | 설명                  |
| --------------- | ------------------- |
| PREDICTED_SCORE | 회귀 모델이 예측한 위험 점수    |
| RISK_GRADE      | low / medium / high |
| RISK_LEVEL      | 기존 코드 호환용 위험 등급     |
| MODEL_TYPE      | regression          |
| MAE             | 모델 MAE              |
| RMSE            | 모델 RMSE             |
| R2_SCORE        | 모델 R2 Score         |
| REQUEST_JSON    | 원본 요청 JSON          |
| RESPONSE_JSON   | 원본 응답 JSON          |

## 9. CONSULTATION_LOGS 테이블

### 설명

3차 LLM, 4차 SLM 상담 기록을 저장한다.

위험 점수 예측 기록이나 이미지 탐지 기록과 연결할 수 있도록 `RISK_LOG_ID`, `DETECTION_ID`를 둔다.

```sql
CREATE TABLE CONSULTATION_LOGS (
    CONSULTATION_ID NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    USER_ID NUMBER,
    RISK_LOG_ID NUMBER,
    DETECTION_ID NUMBER,

    MODEL_TYPE VARCHAR2(30) NOT NULL,
    USER_MESSAGE CLOB NOT NULL,
    ASSISTANT_MESSAGE CLOB NOT NULL,
    CONTEXT_JSON CLOB,

    CREATED_AT TIMESTAMP DEFAULT SYSTIMESTAMP,

    CONSTRAINT FK_CONSULT_USER
        FOREIGN KEY (USER_ID)
        REFERENCES USERS(USER_ID),

    CONSTRAINT FK_CONSULT_RISK
        FOREIGN KEY (RISK_LOG_ID)
        REFERENCES RISK_PREDICTION_LOGS(LOG_ID),

    CONSTRAINT FK_CONSULT_DETECTION
        FOREIGN KEY (DETECTION_ID)
        REFERENCES VISION_DETECTION_LOGS(DETECTION_ID)
);
```

## 13. 조회 SQL 예시

### 전체 위험 점수 예측 기록 조회

```sql
SELECT
    LOG_ID,
    USER_ID,
    SPECIES,
    WEATHER,
    TIME_ZONE,
    PREDICTED_SCORE,
    RISK_GRADE,
    CREATED_AT
FROM RISK_PREDICTION_LOGS
ORDER BY CREATED_AT DESC;
```

### 사용자별 예측 기록 조회

```sql
SELECT
    LOG_ID,
    SPECIES,
    PREDICTED_SCORE,
    RISK_GRADE,
    MESSAGE,
    CREATED_AT
FROM RISK_PREDICTION_LOGS
WHERE USER_ID = 1
ORDER BY CREATED_AT DESC;
```

### high 위험 기록 조회

```sql
SELECT
    LOG_ID,
    USER_ID,
    SPECIES,
    LOCATION_NAME,
    PREDICTED_SCORE,
    RISK_GRADE,
    CREATED_AT
FROM RISK_PREDICTION_LOGS
WHERE RISK_GRADE = 'high'
ORDER BY CREATED_AT DESC;
```

### 위험 점수 70점 이상 조회

```sql
SELECT
    LOG_ID,
    USER_ID,
    SPECIES,
    LOCATION_NAME,
    PREDICTED_SCORE,
    RISK_GRADE,
    CREATED_AT
FROM RISK_PREDICTION_LOGS
WHERE PREDICTED_SCORE >= 70
ORDER BY PREDICTED_SCORE DESC;
```

## 14. Python DB 연동 시 반영 필요 사항

위험 점수 예측 API가 반환하는 주요 필드는 다음과 같다.

```json
{
  "predicted_score": 84.37,
  "risk_score": 84.37,
  "risk_grade": "high",
  "risk_level": "high",
  "model_type": "regression",
  "metrics": {
    "mae": 0.0183,
    "rmse": 0.2324,
    "r2": 0.9992
  }
}
```

따라서 `services/risk_log_service.py`에서는 다음 값을 DB에 저장하도록 수정해야 한다.

| API 필드                      | DB 컬럼                 |
| --------------------------- | --------------------- |
| `input.day`                 | `DAY_TYPE`            |
| `input.camera_type`         | `CAMERA_TYPE`         |
| `input.weather`             | `WEATHER`             |
| `input.location`            | `LOCATION_NAME`       |
| `input.time_zone`           | `TIME_ZONE`           |
| `input.season`              | `SEASON`              |
| `input.species`             | `SPECIES`             |
| `input.object_count`        | `OBJECT_COUNT`        |
| `input.max_bbox_area_ratio` | `MAX_BBOX_AREA_RATIO` |
| `input.avg_bbox_area_ratio` | `AVG_BBOX_AREA_RATIO` |
| `predicted_score`           | `PREDICTED_SCORE`     |
| `risk_grade`                | `RISK_GRADE`          |
| `risk_level`                | `RISK_LEVEL`          |
| `model_type`                | `MODEL_TYPE`          |
| `message`                   | `MESSAGE`             |
| `metrics.mae`               | `MAE`                 |
| `metrics.rmse`              | `RMSE`                |
| `metrics.r2`                | `R2_SCORE`            |

## 16. 주의사항

### 16.1 데이터 및 모델 파일

다음 파일은 DB나 GitHub에 직접 업로드하지 않는다.

```text
data/
models/ml/
*.pkl
*.joblib
```

### 16.2 비밀번호 관리

DB 계정 비밀번호, `.env`, 개인 설정 파일은 GitHub에 업로드하지 않는다.

### 16.3 모델 성능 해석

현재 `risk_score`는 실제 사고 피해 데이터가 아니라 도메인 가중치 기반으로 설계한 점수이다.

따라서 DB에 저장되는 예측 결과는 실제 사고 확률이 아니라, 현재 설계한 위험 점수 체계에 따른 예측 결과로 해석해야 한다.
