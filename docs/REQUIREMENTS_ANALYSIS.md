# WildGuard AI 요구사항 분석서

## 1. 프로젝트 개요

### 프로젝트명

WildGuard AI

### 주제

AI Hub 야생동물 라벨링 JSON 메타데이터 기반 야생동물 위험 점수 예측 및 대응 지원 AI 서비스

### 목적

WildGuard AI는 야생동물 관측 상황을 데이터 기반으로 분석하고, 사용자에게 위험 점수 예측, 이미지 탐지, 대응 상담을 제공하는 AI 서비스이다.

1차 프로젝트에서는 AI Hub 야생동물 라벨링 JSON에서 이미지 메타데이터와 annotation 정보를 추출하여 정형 데이터셋을 구축하고, 야생동물 관측 상황의 위험 점수 `risk_score`를 예측하는 회귀 모델을 구현한다.

| 차수 | 프로젝트 유형       | 핵심 내용                                                              |
| -- | ------------- | ------------------------------------------------------------------ |
| 1차 | 정형 데이터 ML     | AI Hub 라벨링 JSON 메타데이터를 CSV로 변환하고 위험 점수 `risk_score`를 예측하는 회귀 모델 구현 |
| 2차 | Vision / YOLO | 원천 이미지와 bbox 라벨을 활용한 야생동물 탐지                                       |
| 3차 | LLM / RAG     | 위험 점수 예측 결과와 탐지 결과 기반 야생동물 대응 상담                                   |
| 4차 | SLM           | 현장 환경에서 사용할 수 있는 경량 대응 상담 모델                                       |

### 구성 이유

같은 AI Hub 야생동물 데이터셋을 사용하더라도 1차와 2차의 역할을 분리한다.

1차 ML은 이미지 자체를 분석하지 않고, JSON에 포함된 `day`, `camera_type`, `weather`, `location`, `time_zone`, `season`, `species`, `object_count`, `max_bbox_area_ratio`, `avg_bbox_area_ratio` 등의 메타데이터를 정형 데이터로 변환하여 위험 점수 예측에 사용한다.

2차 Vision은 원천 이미지와 bbox annotation을 사용하여 YOLO 기반 객체 탐지 모델을 학습한다.

이렇게 분리하면 1차는 정형 데이터 기반 회귀 모델, 2차는 이미지 기반 객체 탐지 모델로 역할이 명확해진다.

## 2. 문제 정의

야생동물 출현은 농장, 산림, 도로, 마을 주변에서 안전 문제와 재산 피해를 유발할 수 있다. 특히 멧돼지, 고라니, 반달가슴곰과 같은 야생동물은 출현 시간대, 날씨, 객체 수, 접근 거리 등에 따라 위험 수준이 달라질 수 있다.

그러나 일반 사용자가 단순히 야생동물이 찍힌 사진이나 관측 조건만 보고 위험 수준을 수치적으로 판단하기는 어렵다.

| 문제        | 설명                                                   |
| --------- | ---------------------------------------------------- |
| 위험 판단 어려움 | 사용자가 관측 상황의 위험 수준을 직접 판단하기 어렵다.                      |
| 정량적 기준 부족 | 단순히 low / medium / high 등급만으로는 위험 정도를 세밀하게 표현하기 어렵다. |
| 이미지 확인 한계 | 현장 사진만으로 야생동물 객체 여부와 접근 위험을 빠르게 판단하기 어렵다.            |
| 대응 정보 부족  | 동물 종류, 시간대, 날씨, 객체 수에 따른 대응 방법을 알기 어렵다.              |
| 데이터 활용 부족 | AI Hub 라벨링 JSON의 메타데이터가 위험도 판단에 충분히 활용되지 않는다.        |

## 3. 사용자 정의

| 사용자     | 설명                      | 주요 니즈                         |
| ------- | ----------------------- | ----------------------------- |

| 산림관리원   | 산림, 등산로, 보호구역을 관리하는 사용자 | 야생동물 출현 위험 판단, 반복 관측 모니터링     |

## 5. 위험 점수 설계 요구사항

### 5.1 위험 점수 설계 배경

AI Hub 라벨링 JSON은 객체 위치, 동물 종, 촬영 조건 등의 정보는 제공하지만, 실제 사고 피해 점수나 서비스용 위험 점수 라벨은 제공하지 않는다.

따라서 1차 프로젝트에서는 도메인 기준을 바탕으로 위험 점수 `risk_score`를 설계하고, 이 값을 회귀 모델의 예측 대상값으로 사용한다.

### 5.2 위험 점수 반영 요소

| 요소          | 반영 이유                             |
| ----------- | --------------------------------- |
| 동물 종        | 종에 따라 사람, 농작물, 시설물에 미치는 위험도가 다르다. |
| 시간대         | 야간과 새벽은 시야 확보와 현장 대응이 어렵다.        |
| 날씨          | 비, 눈 등은 현장 대응 난도를 높일 수 있다.        |
| 객체 수        | 여러 개체가 동시에 관측되면 위험도가 높아질 수 있다.    |
| bbox 면적 비율  | 화면에서 크게 잡힌 객체는 카메라와 가까울 가능성이 있다.  |
| IR 야간 촬영 여부 | 야간 출현 상황일 가능성이 높다.                |

### 5.3 위험 점수 범위

`risk_score`는 0~100 범위의 숫자로 제한한다.

| 점수 구간        | 등급     |
| ------------ | ------ |
| 0 이상 45 미만   | low    |
| 45 이상 70 미만  | medium |
| 70 이상 100 이하 | high   |

### 5.4 현재 위험 등급 분포

| risk_grade |  count |
| ---------- | -----: |
| medium     | 19,685 |
| high       | 10,343 |
| low        |    672 |

| ---- | ----: |

## 10. 1차 ML 모델 요구사항

### 모델 목적

AI Hub 야생동물 라벨링 JSON에서 추출한 정형 메타데이터를 기반으로 야생동물 관측 상황의 위험 점수 `risk_score`를 예측한다.

### 모델 유형

지도학습 기반 회귀 모델

### 입력 Feature

| 구분  | Feature                                                                       |
| --- | ----------------------------------------------------------------------------- |
| 범주형 | `day`, `camera_type`, `weather`, `location`, `time_zone`, `season`, `species` |
| 수치형 | `object_count`, `max_bbox_area_ratio`, `avg_bbox_area_ratio`                  |

### Target

| Target       | 설명              |
| ------------ | --------------- |
| `risk_score` | 0~100 범위의 위험 점수 |

### 학습 모델

| 항목      | 내용                                     |
| ------- | -------------------------------------- |
| 모델      | RandomForestRegressor                  |
| 저장 파일   | `models/ml/risk_regression_model.pkl`  |
| 학습 스크립트 | `scripts/train_risk_model.py`          |
| 변환 스크립트 | `scripts/convert_aihub_json_to_csv.py` |

### 모델 성능

| 지표       |      값 |
| -------- | -----: |
| MAE      | 0.0183 |
| RMSE     | 0.2324 |
| R2 Score | 0.9992 |

### 성능 해석

현재 `risk_score`는 실제 사고 피해 데이터가 아니라 도메인 가중치 기반으로 설계한 위험 점수이다.

따라서 모델 성능은 실제 현장 사고 위험을 얼마나 정확히 예측했는지가 아니라, 설계된 위험 점수 체계를 회귀 모델이 얼마나 잘 학습했는지를 의미한다.

향후 실제 신고 데이터, 피해 기록, 기상 데이터, 지역 환경 데이터가 확보되면 현재의 `risk_score`를 실제 데이터 기반 라벨로 교체하여 모델을 고도화할 수 있다.

## 11. API 요구사항

### 위험 점수 예측 API

| 항목       | 내용                  |
| -------- | ------------------- |
| Method   | POST                |
| Endpoint | `/api/risk/predict` |
| 입력 형식    | JSON                |
| 출력 형식    | JSON                |

### Request 필수 입력값

| 필드                    | 설명              |
| --------------------- | --------------- |
| `day`                 | 주간 / 야간 촬영 여부   |
| `camera_type`         | RGB / IR 카메라 유형 |
| `weather`             | 날씨              |
| `location`            | 위치              |
| `time_zone`           | 시간대             |
| `season`              | 계절              |
| `species`             | 동물 종            |
| `object_count`        | 객체 수            |
| `max_bbox_area_ratio` | 최대 bbox 면적 비율   |
| `avg_bbox_area_ratio` | 평균 bbox 면적 비율   |

### Response 필수 출력값

| 필드                | 설명                            |
| ----------------- | ----------------------------- |
| `success`         | 요청 성공 여부                      |
| `predicted_score` | 모델이 예측한 위험 점수                 |
| `risk_score`      | `predicted_score`와 동일한 호환용 필드 |
| `risk_grade`      | 점수 기반 위험 등급                   |
| `risk_level`      | 기존 코드 호환용 위험 등급               |
| `model_type`      | 모델 유형                         |
| `message`         | 위험 등급별 대응 메시지                 |
| `metrics`         | 모델 성능 지표                      |
| `input`           | 입력 데이터                        |

## 12. 서비스화 요구사항

### 추가 배경

현재 1차 기능은 위험 점수 회귀 모델과 Flask API를 중심으로 구성되어 있다. 서비스 완성도를 높이기 위해 회원가입, 로그인, 사용자별 예측 기록 저장 기능을 추가한다.

### 인증 요구사항

| ID          | 요구사항       | 설명                                                |
| ----------- | ---------- | ------------------------------------------------- |
| RQ-AUTH-001 | 회원가입       | 사용자는 아이디, 비밀번호, 이름, 소속, 사용자 유형을 입력해 가입할 수 있다.     |
| RQ-AUTH-002 | 비밀번호 해시 저장 | 비밀번호는 평문이 아니라 해시값으로 저장한다.                         |
| RQ-AUTH-003 | 로그인        | 사용자는 아이디와 비밀번호로 로그인할 수 있다.                        |
| RQ-AUTH-004 | 세션 유지      | 로그인 성공 시 세션에 user_id, username, name, role을 저장한다. |
| RQ-AUTH-005 | 로그아웃       | 사용자는 로그아웃으로 세션을 종료할 수 있다.                         |

### 예측 저장 요구사항

| ID          | 요구사항         | 설명                                       |
| ----------- | ------------ | ---------------------------------------- |
| RQ-RISK-001 | 비회원 예측 체험    | 비회원도 `/risk`에서 위험 점수 예측을 실행할 수 있다.       |
| RQ-RISK-002 | 비회원 저장 제한    | 비회원 예측 결과는 DB에 저장하지 않는다.                 |
| RQ-RISK-003 | 회원 예측 저장     | 로그인 사용자의 예측 결과는 Oracle DB에 저장한다.         |
| RQ-RISK-004 | 사용자별 기록 조회   | 로그인 사용자는 본인의 예측 기록만 조회할 수 있다.            |
| RQ-RISK-005 | 관리자 전체 기록 조회 | 관리자와 지자체 담당자는 전체 예측 기록 조회 기능으로 확장할 수 있다. |

### 사용자 역할 요구사항

| role     | 사용자     | 권한                        |
| -------- | ------- | ------------------------- |
| citizen  | 일반 사용자  | 예측 체험, 회원 예측 저장, 내 기록 조회  |
| farmer   | 농장주     | 예측 저장, 내 기록 조회            |
| ranger   | 산림관리원   | 예측 저장, 내 기록 조회            |
| official | 지자체 담당자 | 전체 기록 조회, 대시보드 확장 대상      |
| admin    | 관리자     | 전체 기록 조회, 사용자/통계 관리 확장 대상 |

### 추가 비기능 요구사항

| 구분   | 요구사항                                        |
| ---- | ------------------------------------------- |
| 보안   | 비밀번호는 Werkzeug 기반 해시로 저장한다.                 |
| 개인정보 | `.env`, DB 비밀번호, 원본 데이터는 GitHub에 업로드하지 않는다. |
| 사용성  | 비회원 체험 모드와 회원 저장 모드를 화면에서 명확히 안내한다.         |
| 확장성  | 사용자 role을 기준으로 관리자 페이지와 대시보드로 확장 가능해야 한다.   |
